üéØ MANDATORY Collection Structure

{
  "name": "collection_name",        // REQUIRED: string, alphanumeric + underscore only
  "type": "base",                   // REQUIRED: "base" | "auth" | "view"
  "system": false,                  // REQUIRED: boolean (always false for user collections)
  "listRule": "",                   // REQUIRED: string | null (see API Rules section)
  "viewRule": "",                   // REQUIRED: string | null
  "createRule": null,               // REQUIRED: string | null
  "updateRule": null,               // REQUIRED: string | null
  "deleteRule": null,               // REQUIRED: string | null
  "schema": [                       // REQUIRED: array of field objects
    {
      "name": "field_name",          // REQUIRED: string, valid field name
      "type": "text",                // REQUIRED: valid field type
      "system": false,               // REQUIRED: boolean (always false for user fields)
      "required": true,              // REQUIRED: boolean
      "presentable": true,           // REQUIRED: boolean
      "options": {                   // REQUIRED: object (field-specific options)
        "max": 100,                  // varies by field type
        "min": 1
      }
    }
  ]
}

üîê API Rules Quick Reference

""     // Empty string = PUBLIC ACCESS
null   // null = ADMIN ONLY
"@request.auth.id != ''"  // Authenticated users only

‚ö° Best Practices

  • Always use "schema" array, not "fields"
  • Include "system": false for all user collections and fields
  • Always provide "options": {} even if empty
  • Test with simple collections first, add complexity later
  • Use descriptive field names with underscores
  • Set appropriate rules based on your security needs

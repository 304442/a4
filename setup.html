🚨 THE #1 FAILURE CAUSE: Chicken-and-Egg Rule Problem

❌ WRONG - This FAILS:
{
  "name": "planners",
  "listRule": "@request.auth.id = user_id",  // ❌ user_id doesn't exist yet!
  "schema": [
    {"name": "user_id", "type": "relation", ...}  // ❌ Too late - rules validated first
  ]
}

✅ CORRECT - This WORKS:
{
  "name": "planners", 
  "listRule": "",  // ✅ Start with public/safe rules
  "schema": [
    {"name": "user_id", "type": "relation", ...}  // ✅ Create fields first
  ]
}
// Tool automatically updates to proper rules after fields exist

💡 KEY INSIGHT: PocketBase validates rules BEFORE creating fields!

🔐 API Rules Complete Reference

// 🟢 PUBLIC ACCESS
""                                    // Empty string = anyone can access

// 🔴 ADMIN ONLY  
null                                  // null = admin only access

// 🟡 AUTHENTICATED USERS
"@request.auth.id != ''"              // Any logged-in user

// 🟠 OWNER ONLY (after fields exist!)
"@request.auth.id = user_id"          // User owns the record

// 🔵 VERIFIED USERS
"@request.auth.verified = true"       // Only verified email users

// 🟣 COMPLEX CONDITIONS
"public = true || @request.auth.id = user_id"  // Public OR owner
"@request.auth.role = 'admin'"                 // Role-based access

❌ FATAL ERRORS TO AVOID

1. ❌ Using "fields" instead of "schema"
   "fields": [...]  // BREAKS setup
   "schema": [...]  // WORKS

2. ❌ Missing system property
   { "name": "test" }  // FAILS
   { "name": "test", "system": false }  // WORKS

3. ❌ Missing options object
   { "type": "text" }  // FAILS
   { "type": "text", "options": {} }  // WORKS

4. ❌ Field-dependent rules on creation
   "listRule": "@request.auth.id = user_id"  // FAILS if user_id field doesn't exist
   "listRule": ""  // WORKS, update rules after

5. ❌ String booleans/numbers
   "required": "true"  // FAILS
   "required": true    // WORKS
   "max": "100"       // FAILS  
   "max": 100         // WORKS

6. ❌ Invalid field names
   "field name"       // FAILS (spaces)
   "field-name"       // FAILS (hyphens)
   "field_name"       // WORKS (underscores)

7. ❌ Missing relation properties
   { "type": "relation", "options": { "collectionId": "users" }}  // INCOMPLETE
   { "type": "relation", "options": { 
       "collectionId": "_pb_users_auth_",
       "cascadeDelete": true,
       "minSelect": 1, 
       "maxSelect": 1,
       "displayFields": []
   }}  // COMPLETE

✅ VALIDATION CHECKLIST

COLLECTION LEVEL:
□ name: valid identifier (letters, numbers, underscore)
□ type: exactly "base", "auth", or "view"
□ system: exactly false
□ rules: string or null (not undefined)
□ schema: array (not "fields")

FIELD LEVEL:
□ name: valid identifier, no spaces/hyphens
□ type: valid PocketBase type
□ system: exactly false
□ required: boolean (not string)
□ presentable: boolean (not string)  
□ options: object (not array/null/undefined)

FIELD-SPECIFIC:
□ relation: all 5 required options present
□ select: maxSelect and values present
□ file: maxSelect present
□ text/number: numeric values for min/max
